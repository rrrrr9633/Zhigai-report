# 智改数转诊断报告生成 Skill

## 硬性生成规则

- 唯一有效生成路径：构建 JSON 数据 → 调用 `scripts/run_report.sh` → 由远程授权服务生成 `.docx` → 本地保存最终文件。
- 本地 skill 只作为客户端入口，不得在本地执行核心报告生成逻辑。
- 核心 Python 生成脚本和 docx 模板必须保存在服务端，不随公开分发版 skill 一起交付。
- 禁止生成 Markdown 后转换 Word。
- 禁止新建空白 Word 再写入正文。
- 禁止使用 `create_report_docx.py`、pandoc、Markdown 表格转换、手动复制粘贴式输出作为最终交付。
- 如果远程服务返回授权过期、额度不足或服务端错误，停止并把错误告知用户，不得改用本地绕过方式。

## 核心功能

**远程生成 .docx 文件**：AI 在本地构建结构化 JSON，授权远程服务在服务端加载官方模板并生成完整 Word 报告文档。

### AI 填充的内容

✅ **一、（二）企业总体情况**（300字段落 + 可选三处图片插入）  
✅ **一、（三）企业智改数转情况**（8个小节 + 2个表格）  
✅ **二、企业智改数转痛点需求分析**（智能工厂对标分析结构化填充 + 痛点总述与 6 个痛点小节）  
✅ **三、企业智改数转建设方案**（新版第三部分：三年规划表、总体方案架构、总体技术架构图、建设内容、两年实施计划表、五类项目预期成效、按企业需求动态生成的具体改造项目）

### AI 不处理的内容

❌ 一、（一）企业基本信息表 → 用户手工填写  
❌ 三、（三）服务商推荐 → 已固定内容  
❌ 三、（四）企业诊断后服务计划 → 已固定内容

## 工作流程

### 步骤 1：用户提供素材

用户提供企业调研素材（可以是文本、Word文档或直接对话）：

```
企业基本信息、主营业务、主导产品、生产工艺
企业总体情况相关图片及插入说明（如：第1-3张图片为第一部分插入，第4张为第二部分插入，第5-7张为第三部分插入）
现有设备清单、软件系统清单
网络建设、平台建设、安全措施、AI应用
成熟度评分图、能力要素表图、核心维度对标表图
成熟度评分、企业痛点、改造需求、预算约束
```

### 步骤 2：AI 提取和生成内容

AI 自动：
1. 读取和理解企业素材
2. 按照模板要求生成各部分内容
3. 构建结构化数据（JSON格式）
4. 确保字数、格式、要素完整性

### 步骤 3：调用远程服务生成 .docx

AI 自动调用 `scripts/run_report.sh` 统一入口。该入口只负责把 JSON 数据和本地图片附件上传到远程服务，不在本地读取模板或执行核心生成器：

```bash
bash ~/.codex/skills/zhigai-report/scripts/run_report.sh \
  --license-key ZHIGAI-TRIAL-CHANGE-ME \
  --data 生成的数据.json \
  --output 企业名称_智改数转报告_日期.docx
```

客户端默认使用远程服务 `http://154.201.65.69:8787`；如需切换服务地址，可通过 `--api-base-url`、`ZHIGAI_API_BASE_URL` 或配置文件覆盖。

调用前客户端会通过 `/license/status` 校验授权码。未提供授权码、授权码无效、已停用、已过期或额度用完时会立即停止，并要求提供有效授权码；仅状态为 `active` 时才会上传数据并生成报告。

如果用户已经保存过配置，可以省略 `--api-base-url` 和 `--license-key`：

```bash
bash ~/.codex/skills/zhigai-report/scripts/run_report.sh \
  --data 生成的数据.json \
  --output 企业名称_智改数转报告_日期.docx
```

首次配置可追加 `--save-config`，配置会保存到 `~/.zhigai-report/config.json`。

### 步骤 4：输出最终文件

生成完整的 .docx 文件，包含：
- ✅ 所有文字内容已填充
- ✅ 设备表、软件表、能力要素表、核心维度对标表、三年规划表、两年实施计划表均按结构化数据填充
- ✅ 格式完全保留（字体、行距、边框）
- ⚠️ 企业基本信息表空白（用户手填）
- ✅ 企业总体情况三处图片已按用户指定分组插入（如提供）
- ✅ 第四部分图片可替换“总体技术架构图”占位图，图片段落不使用正文固定行距，避免 Word 中显示不完整

## 核心约束

### 1. 禁止编造

❌ 设备价格、软件价格 → 填"待补充"  
❌ 供应商名称、所在地 → 填"待调研"  
❌ 成熟度评分、成熟度总分、阶段判断、能力要素得分、能力子域得分、核心维度对标结论 → 必须用户明确说明已上传成熟度评分表/评分截图后，逐项对表填写  
❌ 用户未明确说明“已上传成熟度评分表/评分截图”时，不得填写 `智能工厂对标分析`、`成熟度评测`、`工厂要素等级`，也不得根据企业现状推断或补写成熟度内容  
✅ 只有确认用户已上传成熟度评分表后，JSON 中才允许设置 `"成熟度评分表已上传": true` 并填写成熟度相关字段  

### 2. 字数控制

- 企业总体情况：≤300字
- 总体情况：按“主营业务、主导产品、主要工艺、竞争优势、数字化智能化现状与规划”五项填写；合计≤500字
- 设备应用情况：提供整体描述，以及数控/自动化、自动化控制、设备互联、设备监控与决策四项说明
- 网络建设：80-120字
- 每个改造项目：50-100字/项

### 3. 文档格式

- 正文填充内容必须写入真实 Word 段落，不得用 `\n` 塞进同一段落模拟换行。
- 正文填充段落首行缩进 2 字符。
- 表格内脚本填充的数据单元格统一设置为固定值行距 30 磅；模板原有表头、企业基本信息表等未填充区域不得改动。
- 设备表、软件表必须按“关键装备种类 / 关键软件种类”分层逐行展开；每台设备或每套软件独立占一行，同一分类的第一列纵向合并，不得把多项内容堆进一个单元格。

### 4. 企业总体情况图片

- 企业总体情况如有图片，必须使用 `企业总体情况分段` + `企业总体图片`，不要再把全部内容塞进单个 `企业总体情况` 字段。
- `主要产品` 必须优先使用 `主要产品列表`，脚本会自动渲染“（2）主要产品”标题和 `(1)`、`(2)`、`(3)`、`(4)` 分项段落；不要写成一整段连续文字。
- `产品工艺` 必须优先使用 `产品工艺列表`，每项使用 `{ "名称": "产品名称", "流程": "工艺流程" }`，脚本会自动渲染“（3）产品工艺”标题和“编号产品名称 + 下一段流程”。
- 如企业总体情况分段需要自定义标题，可使用 `主要产品标题`、`产品工艺标题`；不得省略产品和工艺两部分的语义标题。
- 产品和工艺清单应保持用户素材顺序；如用户已按一二三四列出，不要合并、压缩或改成顿号串联。
- `第一部分` 图片插入位置：`总体文字` 之后、`主要产品列表` 之前。
- `第二部分` 图片插入位置：`主要产品列表` 之后、`产品工艺列表` 之前。
- `第三部分` 图片插入位置：`产品工艺列表` 之后、`（三）企业智改数转情况` 之前。
- `第四部分` 图片用于替换“三、企业智改数转建设方案”中的“总体技术架构图”占位图；也可直接写入 `智改数转建设方案.总体技术架构图`。
- 第四部分图片必须使用本地可读取的绝对路径；脚本会以图片专用段落插入，不得把图片路径写成正文说明。
- 用户描述“第1-3张图片为第一部分插入，第4张为第二部分插入，第5-7张为第三部分插入，第8张为第四部分架构图”时，AI 必须按上传图片顺序转成四个数组，数组内保持原顺序。
- JSON 中的图片路径必须使用本地可读取的绝对路径；图片文件不存在时应停止并提示缺失文件，不得忽略。

### 5. 企业智改数转建设方案

- 第三部分必须优先使用结构化字段 `智改数转建设方案`；旧字段 `建设思路`、`建设目标`、`规划方案`、`项目实施计划`、`改造项目` 只用于兼容旧数据。
- 第三部分输出不得只写建设思路、目标、三年规划和具体改造项目；`总体方案架构`、`建设内容描述`、`总体实施计划`、`项目预期成效` 四块必须完整填充，否则报告会保留占位内容或结构断裂。
- “建设目标”中的 `总体目标` 必须结合企业痛点、需求分析与建设思路形成完整描述，并覆盖模板中的“内容要求”占位文字；`具体目标.短期`、`具体目标.中期`、`具体目标.长期` 只写纯标题和内容，不得手写 `(1)`、`(2)`、`①`、`②`、`③`；这些编号由模板段落样式自动生成。
- “建设目标”中的短期、中期、长期必须使用结构化对象：`{"时间":"0-8个月，优先合规刚需","内容":"..."}`；括号内是预计实现周期和阶段定位，不得留空。
- “三年智改数转规划方案”后方预留表格必须填入 `三年规划` 数组，不要把表格内容写成普通正文。
- `三年规划[].时间` 写实现所需周期或阶段周期，例如 `0-8个月`、`8-16个月`、`16-24个月`；不得写 `2026年`、`2027年`、`2028年` 这类具体年份。
- 短期、中期、长期阶段可在 `阶段` 字段中写“合规基建建设期 / 业务数据贯通期 / 全链路智能管控期”等阶段定位，时间字段只表达需要多长时间。
- 模板中括号标注“例”的内容仅为参考样例，不是固定输出；生成时必须根据企业行业、现状、预算、痛点和需求动态调整。
- 管理应用层业务系统部署必须按企业需求增删系统。企业不需要 PLM、MES、WMS、ERP、供应商协同等系统时，不得保留对应示例；企业需要其他系统时可加入 QMS、EMS、SCADA、LIMS、CRM、SRM、追溯系统等。
- `总体方案架构` 只控制模板架构图前的 `①` 至 `⑤` 层级名称，不在这些层级标题后插入长正文；各层详细建设说明统一写入后续 `建设内容描述`，避免内容重复并保持架构图原位置。
- `建设内容描述` 必须至少覆盖：基础层基础设施建设、设备与控制层数字化改造、平台层集成中台搭建、管理应用层业务系统部署、智能决策层 BI 与 AI 辅助平台建设。每项 `内容` 至少 100-180 字，并按企业真实场景写清建设范围、改造对象、业务流程和落地边界；必要时用 `子项` 展开 2-4 个应用模块。
- `建设内容描述` 使用数组表达，每项包含 `名称`、`内容`，可选 `子项`；脚本会自动拆成真实 Word 段落。
- `总体实施计划` 必须填入“总体实施计划建议（两年分阶段）”下方表格，固定提供第一、第二、第三阶段三行；每行包含 `阶段`、`周期`、`核心任务`，周期按两年总体窗口合理拆分，不得将表格内容写成普通正文。
- `具体改造项目` 使用数组表达，项目数量和类别不固定，必须围绕企业真实需求生成；不要强行输出模板里的 8 个示例项目。
- 脚本会保留模板中的“具体改造项目”总标题，删除其下方示例项目，并按模板原有三级样式重建：分类使用三级标题编号、项目使用四级标题编号、项目要素使用 `➢` 项目符号。AI 只提供纯文本，不得手写序号或箭头。
- 每个具体改造项目必须包含：项目名称、项目内容、改造环节、建设目标、预计投入、投资回报周期、实施步骤。金额、周期无可靠依据时写“待补充”，不得编造。
- `总体技术架构图` 可写在 `智改数转建设方案.总体技术架构图`，也可写在 `企业总体图片.第四部分`；两者都提供时，以 `智改数转建设方案.总体技术架构图` 为准。
- 提供 `总体技术架构图` 时，脚本按模板原图约 6 英寸宽度替换原架构图，并保留图下方“总体技术架构图”图题；未提供新图时保留模板原架构图。该区域不得出现新旧图片共存，也不得删除图题。
- `总体方案架构` 只输出企业实际涉及的层级，脚本会自动渲染为 `①` 到 `⑤` 序号；缺失或不适用的层级不得保留空标题或示例内容。
- `建设内容描述` 与五类 `项目预期成效` 的小节序号由模板样式保留，AI 只提供无序号的名称与内容，不得在字段值里手写 `(1)`、`①` 等序号。
- `项目预期成效` 必须严格覆盖五类：智能制造成熟度提升、效率提升方面、成本降低方面、合规与质量成效、运营与战略成效。
- `智能制造成熟度提升` 必须拆分为 `智能制造成熟度评分`、`智能工厂梯度培育等级`、`核心能力` 三项；缺少可靠评分依据时不得编造具体分值或等级。
- 效率、成本、合规质量、运营战略四类成效只作定性描述，不体现未经企业确认的具体数字；内容必须与本次建设范围对应。
- `项目预期成效` 必须提供 `项目概括` 或完整 `引导语`；不得保留 `xxx` 占位。推荐使用 `项目概括`，脚本会生成“本项目{项目概括}，本项目实施后预期实现效果如下：”。

### 6. 智能工厂对标分析

- 智能工厂对标分析属于成熟度评分表驱动内容；用户未明确说明已上传成熟度评分表/评分截图时，AI 必须完全跳过该部分，不得生成 `智能工厂对标分析`、`成熟度评测`、`工厂要素等级` 或任何成熟度总分、阶段判断、能力要素得分。
- 只有用户明确说明已上传成熟度评分表/评分截图后，才允许在 JSON 中写入 `"成熟度评分表已上传": true`，并逐项对照评分表填写成熟度相关字段。
- 推荐使用结构化字段 `智能工厂对标分析`，旧字段 `成熟度评测`、`工厂要素等级` 只用于兼容旧数据；两套字段不要同时填充同一内容。
- 用户提供政府系统成熟度评分图（如图1）时，AI 必须读取图中总分、权重、得分和子域得分，并写入 `成熟度总分`、`能力要素分析`、`能力子域分析`；原图必须作为 `成熟度评分图片` 插入在“(2)智能制造成熟度现状评测具体情况”下方、“(3)智能制造成熟度现状分析”上方。
- “(2)智能制造成熟度现状评测具体情况”和“(3)智能制造成熟度现状分析”之间只允许插入成熟度评分表图片，不得写任何说明文字、总分描述、分析段落或表格转写内容。
- 用户提供能力要素维度表图（如图2）时，AI 应将图中“能力要素 / 权重 / 得分 / 现状分析”转换成 `能力要素分析`，填入模板中的能力要素表，不要把截图当正文表格替代。
- 用户提供核心维度对标表图（如图3）时，AI 应将图中“核心维度 / 2025版基础级标准要求 / 企业现状匹配情况”转换成 `核心维度对标分析`，填入模板中的核心维度表。
- `整体等级判定` 必须明确依据：成熟度总分、企业现状、行业特性和《智能工厂梯度培育要素条件（2025年版）》；如果未达到基础级，应明确写“低于基础级”或“数字化萌芽阶段”，不得硬判为基础级。
- `智能工厂梯度培育等级` 用于覆盖“XXXX级智能工厂准入要求对标”中的 `XXXX级`；只有评分材料能够确认基础级、先进级、卓越级或领航级时才填写，不能确认时保留 `XXXX级`。
- `能力子域分析` 必须覆盖：人员域（战略组织、人员技能）、技术域（数据应用、集成、信息安全）、资源域（装备、网络）、制造域（设计、生产、物流、销售、服务）。缺少官方分项得分时写“得分待补充”，不得编造。
- `智能工厂要素分析` 必须覆盖：工厂建设与基础条件、研发设计环节、生产作业环节、生产管理环节、运营管理环节；每项必须拆成“现状”和“差距”。

### 7. 企业痛点分析

- `痛点分析` 必须使用 `总体概述` + `痛点列表` 的结构覆盖模板原占位内容，不得把六项内容合并成一个连续段落。
- `总体概述` 必须结合企业所属行业的质量规范、监管要求和生产经营特点；医疗器械企业重点考虑质量体系、批记录、UDI/追溯、洁净与环境控制、设备验证、供应商与放行等约束，其他行业按其真实规范调整，不得机械套用医疗器械话术。
- `痛点列表` 固定生成 6 项，每项包含 `名称` 与 `内容`；标题必须来自企业真实短板，其中至少一项体现行业特有核心管控短板，不得保留 `XXXXX` 或“痛点分析”占位。

## 技术实现

### 客户端与服务端边界

```text
公开 skill
├── SKILL.md
├── agents/openai.yaml
├── references/
└── scripts/
    ├── run_report.sh
    └── remote_generate.py

私有服务端
└── zhigai-report-server/
    ├── assets/智改数转调研表-20260507v1.0.docx
    ├── scripts/generate_report.py
    ├── server/app.py
    └── requirements.txt
```

- `run_report.sh`：统一客户端入口，只负责调用远程客户端。
- `remote_generate.py`：校验本地输入、上传 JSON 与图片、接收并保存 `.docx`。
- `server/app.py`：校验授权、落盘临时附件并调用生成器。
- `generate_report.py`：只在服务端执行；按原模板文字锚点、表头和图片占位槽原位填充。
- 官方模板：只在服务端保存，公开 skill 不得读取或分发。

图片必须替换原模板槽位：成熟度图占用“智能制造成熟度现状评测具体情况”后的空白段；架构图替换模板自带架构图并删除占位标题。不得按段落下标追加图片。
