# 智改数转诊断报告生成 Skill

## 硬性生成规则

- 唯一有效生成路径：构建 JSON 数据 → 调用 `scripts/run_report.sh` → 由远程授权服务生成 `.docx` → 本地保存最终文件。
- 本地 skill 只作为客户端入口，不得在本地执行核心报告生成逻辑。
- 核心 Python 生成脚本和 docx 模板必须保存在服务端，不随公开分发版 skill 一起交付。
- 禁止生成 Markdown 后转换 Word。
- 禁止新建空白 Word 再写入正文。
- 禁止使用 `create_report_docx.py`、pandoc、Markdown 表格转换、手动复制粘贴式输出作为最终交付。
- 如果远程服务返回授权过期、额度不足或服务端错误，停止并把错误告知用户，不得改用本地绕过方式。

## 核心功能

**远程生成 .docx 文件**：AI 在本地构建结构化 JSON，授权远程服务在服务端加载官方模板并生成完整 Word 报告文档。

### AI 填充的内容

✅ **一、（二）企业总体情况**（总体概述、主营业务、主导产品、主要工艺与设备、竞争优势，建议约 500 字，可按素材完整性适度浮动，可选三处图片）
✅ **一、（三）企业智改数转情况**（200 字以内数字化智能化现状概述 + 设备、网络、平台、软件、安全、AI、其他内容 + 2 个表格）
✅ **二、企业智改数转痛点需求分析**（智能工厂对标分析结构化填充 + 痛点总述与 6 个痛点小节）  
✅ **三、企业智改数转建设方案**（三年规划表、五层总体方案架构、总体技术架构图、分层建设内容、三年实施计划表、五类项目预期成效、按企业需求动态生成的具体改造项目、服务商推荐）

### AI 不处理的内容

❌ 一、（一）企业基本信息表 → 用户手工填写  
❌ 三、（三）服务商推荐 → 已固定内容  
❌ 三、（四）企业诊断后服务计划 → 已固定内容

## 工作流程

### 步骤 1：用户提供素材

用户提供企业调研素材（可以是文本、Word文档或直接对话）：

```
企业基本信息、主营业务、主导产品、生产工艺
企业总体情况相关图片及插入说明（如：第1-3张图片为第一部分插入，第4张为第二部分插入，第5-7张为第三部分插入）
现有设备清单、软件系统清单
网络建设、平台建设、安全措施、AI应用
成熟度评分图、能力要素表图、核心维度对标表图
成熟度评分、企业痛点、改造需求、预算约束
```

### 步骤 2：AI 提取和生成内容

AI 自动：
1. 读取和理解企业素材
2. 按照模板要求生成各部分内容
3. 构建结构化数据（JSON格式）
4. 确保字数、格式、要素完整性

### 步骤 3：调用远程服务生成 .docx

AI 自动调用 `scripts/run_report.sh` 统一入口。该入口只负责把 JSON 数据和本地图片附件上传到远程服务，不在本地读取模板或执行核心生成器：

```bash
bash ~/.codex/skills/zhigai-report/scripts/run_report.sh \
  --license-key ZHIGAI-TRIAL-CHANGE-ME \
  --data 生成的数据.json \
  --output 企业名称_智改数转报告_日期.docx
```

客户端默认使用远程服务 `http://154.201.65.69:8787`；如需切换服务地址，可通过 `--api-base-url`、`ZHIGAI_API_BASE_URL` 或配置文件覆盖。

调用前客户端会通过 `/license/status` 校验授权码。未提供授权码、授权码无效、已停用、已过期或额度用完时会立即停止，并要求提供有效授权码；仅状态为 `active` 时才会上传数据并生成报告。

如果用户已经保存过配置，可以省略 `--api-base-url` 和 `--license-key`：

```bash
bash ~/.codex/skills/zhigai-report/scripts/run_report.sh \
  --data 生成的数据.json \
  --output 企业名称_智改数转报告_日期.docx
```

首次配置可追加 `--save-config`，配置会保存到 `~/.zhigai-report/config.json`。

### 步骤 4：输出最终文件

生成完整的 .docx 文件，包含：
- ✅ 所有文字内容已填充
- ✅ 设备表、软件表、能力要素表、核心维度对标表、三年规划表、三年实施计划表均按结构化数据填充
- ✅ 格式完全保留（字体、行距、边框）
- ⚠️ 企业基本信息表空白（用户手填）
- ✅ 企业总体情况三处图片分别插入总体概述、主导产品、主要工艺与设备之后（如提供）
- ✅ 第四部分图片可替换“总体技术架构图”占位图，图片段落不使用正文固定行距，避免 Word 中显示不完整

## 核心约束

### 1. 禁止编造

❌ 设备价格、软件价格 → 填"待补充"  
❌ 供应商名称、所在地 → 填"待调研"  
❌ 成熟度评分、成熟度总分、阶段判断、能力要素得分、能力子域得分、核心维度对标结论 → 必须用户明确说明已上传成熟度评分表/评分截图后，逐项对表填写  
❌ 用户未明确说明“已上传成熟度评分表/评分截图”时，不得填写 `智能工厂对标分析`、`成熟度评测`、`工厂要素等级`，也不得根据企业现状推断或补写成熟度内容  
✅ 只有确认用户已上传成熟度评分表后，JSON 中才允许设置 `"成熟度评分表已上传": true` 并填写成熟度相关字段  

### 2. 字数控制与写作 Prompt

- `企业总体情况`：严格按“总体概述、主营业务、主导产品、主要工艺与设备、竞争优势”五项生成，建议合计约 500 字；以事实完整、结构清晰为优先，可根据企业素材适度浮动，不得仅因超过 500 字压缩重写、阻断生成或拒绝交付。
- `总体概述`：概括企业介绍、成立与发展基础、企业性质或定位、主要资质、经营规模等已知事实；不写数字化现状，不重复后续四项，素材缺失时不编造。
- `主营业务`：说明企业面向哪些客户或行业提供何种产品、服务或制造能力，突出业务边界和经营模式。
- `主导产品`：保持用户素材中的产品名称、规格和用途顺序；多项产品使用数组，每项独立成段，不压缩为模糊概念。
- `主要工艺与设备`：把产品对应的关键工艺流程、产线和核心设备放在同一作用域描述；优先使用数组，多产品时保留“产品/工艺对象 → 流程与设备”的对应关系。
- `竞争优势`：只基于已知资质、技术积累、客户基础、质量能力、装备能力和交付能力归纳，不使用“行业领先”等无依据判断。
- `企业数字化智能化现状概述`：≤200字，作为`（三）企业智改数转情况`开头独立段落。按“设备数字化基础 → 网络与系统应用 → 数据贯通和智能应用 → 主要短板”组织，只描述现状，不写建设方案、规划目标、成熟度分值或未经证实的效果。
- 设备应用情况：提供整体描述，以及数控/自动化、自动化控制、设备互联、设备监控与决策四项说明。
- 网络建设：80-120字。
- 每个改造项目：50-100字/项。

### 3. 文档格式

- 正文填充内容必须写入真实 Word 段落，不得用 `\n` 塞进同一段落模拟换行。
- 所有新增文字段落（正文、编号条目、项目符号与表格填充数据）统一设置为**固定值行距 30 磅**；模板未填充区域不得改动。
- 图片必须独占图片专用段落。该段落设置为单倍行距或最小值行距，**禁止**固定值 30 磅；企业总体图片、成熟度评分图片、总体技术架构图均适用。
- 图片只限制最大宽度，按原始比例计算高度；不得为凑版面强制高度，也不得将图片路径写入正文。
- 设备表、软件表必须按“关键装备种类 / 关键软件种类”分层逐行展开；每台设备或每套软件独立占一行，同一分类的第一列纵向合并，不得把多项内容堆进一个单元格。

### 4. 企业总体情况与图片

- 新数据必须使用结构化对象 `企业总体情况`，字段固定为 `总体概述`、`主营业务`、`主导产品`、`主要工艺与设备`、`竞争优势`；`企业总体情况分段` 和 `总体情况` 仅用于兼容旧数据，新报告不得继续生成旧结构。
- `主导产品` 可使用字符串或数组；多项产品优先使用数组，每项独立成段，保持用户素材顺序，AI 不得手写序号。
- `主要工艺与设备` 可使用字符串或数组；需要表达多种产品工艺时，每项使用 `{ "名称": "产品或工艺对象", "流程": "关键流程与核心设备" }`，名称与内容分别独立成段。
- 产品、工艺和设备清单必须保持用户素材顺序；用户已按一二三四列出时，不得合并、压缩或改成顿号串联。
- `第一部分` 图片插入位置：`总体概述`全部文字之后、`主营业务`之前。
- `第二部分` 图片插入位置：`主导产品`全部文字之后、`主要工艺与设备`之前。
- `第三部分` 图片插入位置：`主要工艺与设备`全部文字之后、`竞争优势`之前。
- 三处图片只改变插入位置，图片分组内的内容和顺序保持不变。
- `第四部分` 图片用于替换“三、企业智改数转建设方案”中的“总体技术架构图”占位图；也可直接写入 `智改数转建设方案.总体技术架构图`。
- 第四部分图片必须使用本地可读取的绝对路径；脚本会以图片专用段落插入，不得把图片路径写成正文说明。
- 用户描述“第1-3张图片为第一部分插入，第4张为第二部分插入，第5-7张为第三部分插入，第8张为第四部分架构图”时，AI 必须按上传图片顺序转成四个数组，数组内保持原顺序。
- JSON 中的图片路径必须使用本地可读取的绝对路径；图片文件不存在时应停止并提示缺失文件，不得忽略。

### 5. 企业智改数转建设方案

- 第三部分必须优先使用结构化字段 `智改数转建设方案`；旧字段 `建设思路`、`建设目标`、`规划方案`、`项目实施计划`、`改造项目` 只用于兼容旧数据。
- 第三部分输出不得只写建设思路、目标、三年规划和具体改造项目；`总体方案架构`、`建设内容描述`、`总体实施计划`、`项目预期成效`、`服务商推荐` 五块必须完整填充，否则报告会保留占位内容或结构断裂。
- “建设目标”中的 `总体目标` 必须结合企业痛点、需求分析与建设思路形成完整描述，并覆盖模板中的“内容要求”占位文字；`具体目标.短期`、`具体目标.中期`、`具体目标.长期` 只写纯标题和内容，不得手写 `(1)`、`(2)`、`①`、`②`、`③`；这些编号由模板段落样式自动生成。
- “建设目标”中的短期、中期、长期必须使用结构化对象：`{"时间":"0-12个月，优先合规刚需","内容":"..."}`；括号内是预计实现周期和阶段定位，不得留空。
- “三年智改数转规划方案”后方预留表格必须填入 `三年规划` 数组；`两年规划` 仅用于读取旧数据，新报告不得生成。三年规划默认按 0-12 个月、13-24 个月、25-36 个月三个实施窗口填写，不要把表格内容写成普通正文。
- `三年规划` 每一行必须包含 `阶段`、`时间`、`核心任务`、`关键指标` 或 `关键成果`；模板表格四列必须全部有实质内容。
- 规划表的每个阶段必须写清实施范围、关键任务、阶段成果/关键指标，禁止只填写阶段名称或留空。
- 模板中括号标注“例”的内容仅为参考样例，不是固定输出；生成时必须根据企业行业、现状、预算、痛点和需求动态调整。
- 管理应用层业务系统部署必须按企业需求增删系统。企业不需要 PLM、MES、WMS、ERP、供应商协同等系统时，不得保留对应示例；企业需要其他系统时可加入 QMS、EMS、SCADA、LIMS、CRM、SRM、设备管理或质量数据管理系统等。
- `总体方案架构` 每个层级必须同时填写 `层级` 和 `概述`。服务端会在模板 `①` 至 `⑤` 层级标题后插入概述正文；概述应说明该层的建设对象、职责、与上下层的数据流向及对企业痛点的作用，不能只输出层级名称。
- `建设内容描述` 必须严格输出五层且不得缺层：基础层（车间工业网络、网络与数据安全、计算与存储资源）、设备与控制层（关键设备联网改造、生产与工艺数据采集、设备状态监控与运维）、平台层（IIoT、企业数据中台及按行业需要配置的协同平台）、业务应用层（按需说明 ERP、MES、QMS 等）、智能决策层（生产、质量、设备、仓储物流、管理层看板及 AI）。每项必须包含非空的 `建设内容`、`建设效果`、`解决痛点` 和 `子项`；名称可使用完整标题或层级简称，服务端按层级职责归一映射。即使某层需要弱化，也必须结合企业现状说明本阶段采用的轻量化建设、暂缓边界或基础准备，不得省略该层、输出空白或使用“补充内容”“待补充”“...”等占位文字。
- `总体实施计划` 必须填入“总体实施计划建议（三年分阶段）”下方表格，固定提供第一、第二、第三阶段三行；每行包含 `阶段`、`周期`、`核心任务`，周期按三年总体窗口拆分为 0-12 个月、13-24 个月、25-36 个月，不得将表格内容写成普通正文。
- `具体改造项目` 使用数组表达，项目数量和类别不固定，必须围绕企业真实需求生成；不要强行输出模板里的 8 个示例项目。
- 脚本会保留模板中的“具体改造项目”总标题，删除其下方示例项目，并按模板原有三级样式重建：分类使用三级标题编号、项目使用四级标题编号、项目要素使用 `➢` 项目符号。AI 只提供纯文本，不得手写序号或箭头。
- `具体改造项目` 必须按实际需要归入网络与基础设施、智能装备与设备联网、工业软件应用、平台与数据中台、人工智能与特色应用、智能决策与可视化六类；每个项目必须包含：项目内容、改造环节、建设目标、预计投入、投资回报周期、实施步骤。项目建设目标只写可验收的功能、范围和管理结果，不体现成熟度提升分值或未经基线测量的效率、成本降幅。
- `预计投入` 和 `投资回报周期` 必须给出可执行的数字估算，不得填写“待补充”“待调研”“待评估”。`预计投入` 使用“XX万元”或“XX-XX万元（估算）”，`投资回报周期` 使用“XX个月”或“XX-XX个月（估算）”；估算应结合设备数量、接口复杂度、软件模块、部署方式、保密要求、实施服务和企业规模，不能把估算数值描述成供应商正式报价或已确认收益。
- `总体技术架构图` 可写在 `智改数转建设方案.总体技术架构图`，也可写在 `企业总体图片.第四部分`；两者都提供时，以 `智改数转建设方案.总体技术架构图` 为准。
- 提供 `总体技术架构图` 时，脚本按模板原图约 6 英寸宽度替换原架构图，并保留图下方“总体技术架构图”图题；未提供新图时保留模板原架构图。该区域不得出现新旧图片共存，也不得删除图题。
- `总体方案架构` 只输出企业实际涉及的层级，脚本会自动渲染为 `①` 到 `⑤` 序号；缺失或不适用的层级不得保留空标题或示例内容。

- `项目预期成效` 必须严格覆盖五类：智能制造成熟度提升、效率提升方面、成本降低方面、合规与质量成效、运营与战略成效。`项目概括` 建议 100-150 字；成熟度提升下的三个子项各写 80-120 字；效率、成本、合规质量、运营战略四类各写 150-220 字，至少分别说明建设动作、作用机制、业务影响和后续支撑，不能只写一句结果判断。
- `智能制造成熟度提升` 必须拆分为 `智能制造成熟度评分`、`智能工厂梯度培育等级`、`核心能力` 三项；缺少可靠评分依据时不得编造具体分值或等级。
- 效率、成本、合规质量、运营战略四类成效只作定性描述，不体现未经企业确认的具体数字；内容必须与本次建设范围对应。
- `项目预期成效` 必须提供 `项目概括` 或完整 `引导语`；不得保留 `xxx` 占位。推荐使用 `项目概括`，脚本会生成“本项目{项目概括}，本项目实施后预期实现效果如下：”。

### 5.1 建设思路决策矩阵

建设思路必须先判断企业成熟度评分相对当前等级区间的位置：处于等级下限附近时，对标同级标准上限规划；处于等级上限附近时，按成熟度评分提升 0.4 分的建设目标规划。优先级按以下顺序组织：

1. 设备自动化程度低：优先设备数控化或自动化升级，再安排联网和系统建设。
2. 订单饱和：优先支撑产能、交付、质量和现场协同的数字化升级。
3. 订单不饱和或存在产能过剩：优先优化管理流程、管理标准，补充轻量化数字化系统，避免盲目扩充产能。
4. 成熟度评分为三级：在设备、系统和数据基础上增加数字化升级与 AI 应用赋能。

无法确认成熟度评分、订单状态或设备现状时，必须明确写“待调研/待确认”，不得擅自选择建设路径。

### 5.2 三年规划与五层架构

- 规划周期固定为三年，新报告使用 `三年规划` 和 `总体实施计划`；`两年规划` 仅用于读取旧数据，不得作为新版输出。短期、中期、长期及实施计划统一采用 0-12 个月、13-24 个月、25-36 个月三个窗口。
- `总体方案架构` 固定按自下而上五层输出：基础层、设备与控制层、平台层、业务应用层、智能决策层；每层必须有针对企业痛点、建设目标和数据流向的概述。

```text
公开 skill
├── SKILL.md

```

- `run_report.sh`：统一客户端入口，只负责调用远程客户端。
- `remote_generate.py`：校验本地输入、上传 JSON 与图片、接收并保存 `.docx`。
- `server/app.py`：校验授权、落盘临时附件并调用生成器。
- `generate_report.py`：只在服务端执行；按原模板文字锚点、表头和图片占位槽原位填充。
- 官方模板：只在服务端保存，公开 skill 不得读取或分发。

图片必须替换原模板槽位：成熟度图占用“智能制造成熟度现状评测具体情况”后的空白段；架构图替换模板自带架构图并删除占位标题。不得按段落下标追加图片。
